To test your supervisor-worker code thoroughly, you need to simulate four specific scenarios: Normal Operation, Worker Exit, Worker Crash, and Supervisor Shutdown.
Compile your code first: gcc -Wall -o supervisor supervisor.c.
## 1. Test Normal Output Redirection
Verify that the supervisor correctly captures and prints the worker's output through the pipe.

* Command: ./supervisor ls -l
* Expected Result: You should see [Supervisor] Worker started..., followed by the file list prefixed with [Worker]:, and then [Supervisor] Worker stopped. Restarting.... [1, 2] 

## 2. Test Auto-Restart (Worker Exit)
Confirm that when the worker finishes its task, the supervisor automatically spawns a new one. [3] 

* Command: ./supervisor echo "Hello"
* Expected Result: The program should print "Hello", say it is restarting, and then print "Hello" again every second in an infinite loop. [4] 

## 3. Test Worker Crash/Failure
Verify the code handles a command that doesn't exist or crashes.

* Command: ./supervisor non_existent_command
* Expected Result: You should see Exec failed: No such file or directory. The supervisor should acknowledge the stop and attempt to restart it every second. [5, 6] 

## 4. Test Supervisor Shutdown (SIGINT)
Ensure that hitting Ctrl+C cleans up the worker process before the supervisor exits, preventing "orphan" workers. [7, 8] 

* Steps:
1. Run ./supervisor sleep 100.
   2. Open a second terminal and run ps -ef | grep sleep to find the child’s PID.
   3. Go back to the first terminal and press Ctrl+C.
* Expected Result: The supervisor should print [Supervisor] Shutting down... and specifically mention the child's PID. [4] 
* Verification: Run ps -ef | grep sleep again in the second terminal; the sleep process should be gone. [9, 10] 

## Summary of Edge Cases to Watch

| Scenario [11, 12] | What to look for |
|---|---|
| Zombie Processes | Run top or ps after the worker finishes. If you see <defunct>, your waitpid is working correctly but needs to be called promptly. |
| Buffer Overflow | If the worker outputs more than 256 bytes at once, ensure the while loop in the supervisor continues reading without crashing. |

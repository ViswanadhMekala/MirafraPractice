## Detailed Notes: C, Booting, and Startup
------------------------------
## 1. Weak Handlers in C

* Definition: A "weak" function is a default implementation that can be overridden by the user.
* How it works: Compilers (like GCC) use __attribute__((weak)). If the linker finds two functions with the same name, it normally throws an error. However, if one is "weak," the linker prioritizes the "strong" one and ignores the weak one.
* Practical Example:

// In library.cvoid __attribute__((weak)) Interrupt_Handler(void) {
    while(1); // Default: do nothing/hang
}
// In main.c (User code)void Interrupt_Handler(void) {
    printf("UART Data Received!"); // User's custom logic
}


------------------------------
## 2. Booting vs. Startup Code

* Booting (The "Getting to the App" stage):
* Goal: Wake up hardware and find the software.
   * Need: CPUs are "dumb" at power-on. They don't know where the OS is or how to talk to RAM. The boot process initializes the clock, checks the hardware (POST), and loads the code into execution memory.
* Startup Code (The "Preparing C" stage):
* Goal: Setup the C Runtime.
   * Need: C programs expect global variables to have values and the "stack" to be ready. Startup code (usually startup.s) handles this.
   * Tasks:
   1. Set Stack Pointer.
      2. Copy .data (initialized globals) from Flash to RAM.
      3. Zero out .bss (uninitialized globals).
      4. Jump to main().
   
------------------------------
## 3. ARM Cortex-M Hardware Boot Procedure
Unlike other CPUs, ARM Cortex-M has a hardwired boot sequence:

   1. Read Address 0x00000000: The CPU automatically loads the value here into the Main Stack Pointer (MSP).
   2. Read Address 0x00000004: The CPU loads the value here (the Reset Vector) into the Program Counter (PC).
   3. Execute: The CPU starts running instructions at the address pointed to by the PC (usually the Reset_Handler).

------------------------------
## 4. Booting from eMMC (Non-XIP)

* The Problem: eMMC is like a hard drive; the CPU cannot "Execute-In-Place" (XIP) from it.
* The Solution (Multi-Stage):
* Stage 1 (ROM): A tiny program inside the CPU silicon reads a small block from eMMC into internal SRAM.
   * Stage 2 (SPL): This "Secondary Program Loader" initializes the large external DDR RAM.
   * Stage 3 (Application): The SPL copies the entire application/OS from eMMC to the DDR RAM and jumps to it.

------------------------------
## 5. Dual Booting & Linker Scripts
To have two apps in one internal Flash, you must "partition" the memory.

* The Linker Script (.ld): Defines two regions: FLASH_APP1 (e.g., 0x08000000) and FLASH_APP2 (e.g., 0x08040000).
* The Bootloader Jump: To switch apps, the bootloader must:
1. Update the VTOR (Vector Table Offset Register) to the new app's address.
   2. Set the MSP (Stack Pointer) to the first word of the new app.
   3. Jump to the Reset_Handler address (second word of the app).

------------------------------
